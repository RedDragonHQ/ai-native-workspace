# AI-Native Workspace 架构蓝图

> **本文档是给你的(AI Agent)执行指令。**
> 读完本文档后,按照「初始化步骤」章节操作,为用户建立一个 AI-native 工作空间。
> 架构描述是"为什么这样设计",初始化步骤是"你该做什么"。

---

## 一、四层架构

```
┌─────────────────────────────────────────────────────┐
│  应用层 (Application)                                │
│  具体项目 + 知识库 + 收件箱 + 工作流                    │
├─────────────────────────────────────────────────────┤
│  协议层 (Protocol)                                   │
│  CLAUDE.md 约定 · CLI 工具 · MCP Servers             │
├─────────────────────────────────────────────────────┤
│  智能体层 (Agent)                                     │
│  Claude Code · Codex · Cursor · 任意能读 md 的 agent  │
├─────────────────────────────────────────────────────┤
│  模型层 (Model)                                      │
│  DeepSeek · Claude · Qwen · OpenAI · 本地模型          │
└─────────────────────────────────────────────────────┘
```

### 1.1 模型层 (Model)

Agent 背后的 LLM 推理能力。**关键设计:用统一协议对接多个提供商,随时可切换。**

| 概念       | 说明                                                                                                |
| -------- | ------------------------------------------------------------------------------------------------- |
| **统一协议** | 推荐 Anthropic Messages API 兼容协议作为标准接口。DeepSeek、Qwen、第三方中转都支持 `base_url` override,Agent 不需要改代码就能换模型 |
| **多提供商** | 至少准备 2 个:一个日常低成本(如 DeepSeek),一个强推理备用(如 Claude via 中转)。单一提供商是风险点                                   |
| **环境隔离** | 不同项目/场景用不同环境变量脚本(如 `source daily.sh` / `source strong.sh`),切换只需重新 source                          |
| **代理链**  | 部分 API(如 OpenRouter)可能有地域限制,需要 SOCKS5/HTTP 代理链。代理链是基础设施,不是 Agent 该操心的事                            |

**你不需要在初始化时配置模型层** —— 用户既然能让你读取这份文档,说明已有自己的 API key 和/或接入方式。你只需要知道模型层存在,Agent 通过它获得推理能力。

### 1.2 智能体层 (Agent)

**一个 Agent 就够了。** 不要一开始就搞 multi-agent。

| 概念 | 说明 |
|------|------|
| **CLAUDE.md 分层认知** | Agent 的运行时配置文件,不是普通 README。分层自动加载:全局 → 工作区根 → 子项目 → 持久记忆 |
| **渐进式上下文** | Agent 的上下文窗口有限。不要全量读文件,先看索引/摘要,按需深入 |
| **工具调用** | Agent 通过 MCP server 和 CLI 获得执行能力(文件操作、搜索、Git、Docker 等) |
| **状态持久化** | 跨会话的关键信息写入 CLAUDE.md 或项目文档,下次启动自动加载 |

**CLAUDE.md 分层加载协议**:

```
~/.claude/CLAUDE.md              ← 全局(Agent 身份、通用约定)
<workspace>/CLAUDE.md            ← 工作区根(目录树、项目清单、跨项目工作流)
<workspace>/<project>/CLAUDE.md  ← 子项目级(cd 进去时自动追加加载)
```

**关键原则**:
- CLAUDE.md 是 **Agent 的操作系统配置**。优先为 agent 消费优化,但人类完全能读 —— 当 agent 不可用时,人类可以用同样的协议接手操作。
- 每个 CLAUDE.md 职责不同:根级管全局认知,子项目级管技术细节
- 跨会话不失效的信息(用户偏好、架构决策、踩坑教训)写在 CLAUDE.md 或项目文档里,不是工作日志

### 1.3 协议层 (Protocol)

Agent 和外部世界交互的通道。**三类协议,能力递增**:

```
文件协议          CLI 工具           MCP Servers
(CLAUDE.md 约定)  (git/curl/docker)  (搜索/数据库/API)
     ↓                ↓                  ↓
  Agent 的工作手册   Agent 的双手       Agent 的专业工具
```

| 协议类型 | 说明 | 初始化时需要 |
|---------|------|:-----------:|
| **文件协议** | CLAUDE.md 约定(目录结构、命名规范、工作流)| ✅ 必须 |
| **CLI 工具** | git、curl、docker、node、python 等系统工具 | ✅ 用户环境自带 |
| **MCP Server** | 搜索引擎、数据库、外部 API 桥接 | ⬜ 按需加 |

**MCP Server 是扩展能力,不是必需品。** 初始化时不需要配置任何 MCP。用户后续按需接入(如搜索、Wiki 语义搜索、外部 API 桥接)。

**文件协议是核心** —— CLAUDE.md 本身就是最重要的协议。它定义了 Agent 如何理解工作区、如何操作文件、如何管理知识。

### 1.4 应用层 (Application)

实际的工作内容。**三个核心组件**:

```
workspace/
├── CLAUDE.md              ← 工作区认知入口
├── inbox/                 ← 信息缓冲区(会话收口、TODO、临时笔记)
│   ├── README.md          ← inbox 格式约定
│   └── TODO.md            ← 跨会话流动清单
├── projects/              ← 具体项目(或直接在根目录下)
│   ├── project-a/
│   │   └── CLAUDE.md      ← 项目级认知
│   └── project-b/
│       └── CLAUDE.md
└── wiki/                  ← 知识库(可选,成熟后再建)
    └── MANIFEST.md        ← 渐进式加载索引
```

| 组件         | 角色                        |  初始化时  |
| ------------ | ------------------------- | :----: |
| **inbox/** | 信息缓冲区:会话摘要、临时笔记、流动 TODO   |  ✅ 必须  |
| **项目文件夹**  | 每个项目一个文件夹 + 自己的 CLAUDE.md | ✅ 扫描已有 |
| **wiki/**  | 结构化知识库(分级文档 + 渐进式索引)      |  ⬜ 可选  |

**inbox/ 是核心创新点**:
- 它是 **草稿区**,不是最终产出。信息先进 inbox,成熟后再沉淀到 wiki 或项目文档
- 每次会话结束写一个 inbox 文件(摘要 + 产出 + 要点),是 Agent 的"工作日志"
- `TODO.md` 是跨会话的唯一待办清单,Agent 启动时先读它,知道"上次做到哪了"
- inbox 降低认知负担:不用当场决定信息该放哪,先丢 inbox,以后整理

**wiki/ 是进阶能力**:
- 当 inbox 积累到一定量,开始需要结构化知识时,再建 wiki
- 核心机制是 **渐进式加载**:`MANIFEST.md` 是索引(每篇一行摘要),Agent 先看索引再按需 Read 详情
- 文档可以带成熟度分级(`raw` → `distilled` → `canonical`)和时效分级(`active` → `sleeping` → `deprecated`)。这些来自 PKM 实践 —— 如果你后续想深入 PKM 体系,框架留了扩展空间。

## 二、初始化步骤

**你是 AI Agent。现在请按照以下步骤为用户初始化 workspace。**

### 前置:确认工作区

```
1. 确认当前目录是用户指定的 workspace 根目录
2. 检查是否已有 CLAUDE.md(有 → 增量更新;没有 → 全新创建)
3. 检查是否已有 .claude/ 或 .git/(了解现有配置状态)
```

### Step 1:扫描工作区

```
扫描根目录下所有子文件夹,生成目录树。对每个子文件夹:
- 识别是否为"项目"(有代码文件 / 有 README / 有 package.json / 有 .git 等)
- 识别是否已有 CLAUDE.md
- 生成一行摘要(基于文件夹名和内容推断)

扫描根目录下的文件:
- 识别脚本文件(.sh / .py 等)
- 识别配置文件(.json / .yml / .env 等)
- 忽略隐藏目录(.git / .venv / node_modules 等)
```

**产出**:一份结构化的工作区地图,用于生成 CLAUDE.md。

### Step 2:生成根 CLAUDE.md

```markdown
# CLAUDE.md — <workspace 名称>

> 为 AI Agent 在此工作区工作时提供指导。每次启动自动加载。

## 目录结构

<Step 1 扫描结果,用 tree 格式,每个目录/文件一行注释>

## 工作流

### 会话结束(必做)
1. 写 inbox 文件:`inbox/YYYY-MM-DD-简短描述.md`
   - 格式:标签 + 摘要 + 成就列表 + 产出文件路径
2. 更新 `inbox/TODO.md`(如有新增/完成项)

### 会话恢复
1. 读 `inbox/TODO.md` 的"下次进来从这里开始"段
2. 按需读 inbox/ 近期文件了解上下文

### 知识沉淀路径
inbox/(草稿)→ 项目文档 或 wiki/(成熟知识)

## 协作原则

- CLAUDE.md 是 Agent 的运行时配置,改动后下次启动生效
- 不要全量读大文件,先看摘要/索引,按需深入
- 技术验证 > 社交安抚:收到反馈先验证事实,再决定实施或反驳
- inbox/TODO.md 是草稿,引用时必须先校验路径和分类是否正确
```

**注意**:
- 保持精简,不超过 100 行。详细技术细节放在子项目 CLAUDE.md
- 不要预写具体的技术栈/工具列表 —— 这些在子项目级别或用户后续补充
- 工作流约定是最核心的部分,它定义了 Agent 的日常行为模式

### Step 3:创建 inbox 结构

**3a. 创建 `inbox/` 目录 + `inbox/README.md`**:

```markdown
# inbox/ — 信息缓冲区

## 文件格式

文件名:`YYYY-MM-DD-简短描述.md`

文件内容:
---
tags: [相关标签]
date: YYYY-MM-DD
---

# 标题

## 摘要
一段话概括本次工作/发现。

## 成就
- 做了什么 1
- 做了什么 2

## 产出文件
- `path/to/file1` — 描述
- `path/to/file2` — 描述

## 对话要点
(可选)关键讨论和决策。
```

**3b. 创建 `inbox/TODO.md`**:

```markdown
# TODO — 跨会话流动清单

> Agent 每次启动先读本文件的「下次进来从这里开始」段。
> 完成项移到底部「✅ 已完成」区。

## 🔄 下次进来从这里开始

(当前最重要的待办,1-3 项)

## 📋 进行中

## 📋 待办

## ✅ 已完成
```

### Step 4:(可选)为已有项目生成子 CLAUDE.md

对 Step 1 识别出的每个项目文件夹,如果还没有 CLAUDE.md:

```markdown
# CLAUDE.md — <项目名>

> 为 Agent 在本项目工作时提供指导。cd 进来时自动追加加载。

## 项目概述
<基于扫描结果的一行摘要>

## 目录结构
<该项目的 tree>

## 关键文件
<列出最重要的 3-5 个文件及其作用>

## 启动/运行
<如果能从 package.json / Makefile / README 推断>
```

**注意**:子项目 CLAUDE.md 只写骨架,让用户和 Agent 在日常使用中逐步丰富。不要试图一次写完美。

### Step 5:验证

```
1. 检查 CLAUDE.md 是否语法正确、结构清晰
2. 检查 inbox/ 目录和两个文件是否创建成功
3. 如果有子项目 CLAUDE.md,检查每个是否合理
4. 输出初始化摘要,告知用户下一步可以做什么
```

## 三、日常工作流约定

### 3.1 会话生命周期

```
启动 → 读 CLAUDE.md + TODO.md
  → 用户提需求
  → Agent 工作(读/写文件、调工具)
  → 会话结束:
     1. 写 inbox/YYYY-MM-DD-xxx.md(工作摘要)
     2. 更新 inbox/TODO.md(新增/完成/调整待办)
```

### 3.2 知识沉淀路径

```
信息产生 → inbox/(草稿,低门槛,随时丢进去)
         → 成熟后升级到:
            - 项目文档(项目特有的技术细节)
            - wiki/(跨项目的通用知识)
         → 过时后降级:
            - freshness 改 sleeping/deprecated
            - 或移入 archive/
```

**inbox 的价值**:降低写文档的心理门槛。不是每件事都值得写正式文档,但每件事都值得记一笔。inbox 就是那个"随便记"的地方。

### 3.3 CLAUDE.md 维护原则

| 何时更新 | 谁更新 | 更新什么 |
|---------|--------|---------|
| 新增子项目 | Agent(初始化时) | 根 CLAUDE.md 目录树 + 子项目 CLAUDE.md |
| 工作流变更 | 用户决策,Agent 执行 | 根 CLAUDE.md 工作流段 |
| 踩坑/架构变更 | Agent(会话中) | 对应项目的 CLAUDE.md |
| 用户偏好/反馈 | Agent(收到反馈时) | CLAUDE.md 或项目文档 |

**铁律**:CLAUDE.md 保持精简。超过 150 行说明该拆分到子项目或 wiki。

### 3.4 反模式(不要做)

| ❌ 不要做 | ✅ 应该做 |
|----------|----------|
| 一开始就搞 multi-agent | 单 agent 起步,复杂了再拆 |
| 全量读工作区所有文件 | 先看 CLAUDE.md 索引,按需深入 |
| 跳过 inbox 直接写 wiki | inbox 是缓冲区,wiki 是沉淀层 |
| CLAUDE.md 写 500 行 | 精简到 100 行内,细节放子项目 |
| 把 TODO 当指令无脑执行 | TODO 是草稿,引用前必须校验 |
| 自动触发写 inbox 文件 | 可以提示用户,但不要不经确认就写 |
| 预设每个项目的最佳实践 | 先观察用户的工作方式,再建议 |

## 四、扩展路径(初始化后按需升级)

以下是 workspace 成熟后的可选升级,**初始化时不要做**:

| 升级 | 触发条件 | 说明 |
|------|---------|------|
| **wiki/ 知识库** | inbox 积累 20+ 文件,开始需要结构化 | 建 wiki/ + MANIFEST.md 索引 + 渐进式加载 |
| **MCP Server** | 需要搜索引擎/数据库/外部 API | 接入 SearXNG / 语义搜索 / 专业工具 |
| **多模型路由** | 单一模型不够用或成本太高 | 多 LLM 提供商 + 环境变量切换脚本 |
| **Homepage 仪表盘** | 想可视化工作记录和项目进度 | 静态 HTML + worklog.js 数据 |
| **自托管服务** | 需要 RSS/自动化/搜索等基础设施 | Docker 部署 SearXNG / Miniflux / n8n |
| **自动化脚本** | 手动操作太频繁 | build_manifest.py / rss_digest.py 等 |
| **Multi-Agent** | 单 agent 上下文窗口确实不够 | 拆分专项 agent,用工作流编排 |

---

## 五、给 Agent 的最后提醒

1. **你是工具,不是主角。** 用户做决策,你做执行。不确定就问,不要自作主张。
2. **CLAUDE.md 是你的操作系统。** 每次启动先读它,它就是你的行为规范。
3. **inbox 是你的工作日志。** 每次会话结束写一笔,这是你跨会话连续性的桥梁。
4. **渐进式加载是你的生存策略。** 上下文窗口有限,先看索引再看详情,永远不要全量读。
5. **简单比完美更重要。** 一个能用的 inbox 比一个完美的 wiki 有价值一百倍。
